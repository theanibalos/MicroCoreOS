# MicroCoreOS Architecture Skill
    
This skill enforces the modular, asynchronous, and resilient design principles of MicroCoreOS.

## 🏛️ The Three Golden Rules (SACRED)
These rules are inviolable. Any deviation is a violation of the "Atomic Microkernel" architecture.

1.  **Rule 1: The Core is Sacred**: Files in `/core` MUST NOT be modified for business logic or observability.
2.  **Rule 2: Absolute Tool Isolation**: Tools are independent atoms. A Tool **MUST NEVER** import, request, or depend on another Tool. They are 100% agnostic of each other.
3.  **Rule 3: Plugin-as-Glue (The Bridge)**: Plugins are the ONLY place where Tools collaborate. If two Tools need to work together (e.g., HTTP needs Identity), the Plugin MUST orchestrate the bridge.

> [!IMPORTANT]
> **Tool vs Plugin Philosophy**: 
> - **Tools** provide technical *Capabilities* (e.g., "I know how to send HTTP packets"). 
> - **Plugins** provide business *Intent* (e.g., "I want to create a user using HTTP and SQL").

## 🏗️ Development Rules

### Environment & Execution
1.  **Mandatory Runner**: ALWAYS use `uv run main.py`. Do not invoke `python` directly.
2.  **Manifest Generation**: The system regenerates `AI_CONTEXT.md` on every boot. Ensure the system runs at least once after architectural changes.

### Plugins & Models
1.  **Isolation**: Communication between domains MUST be via `event_bus`. No cross-domain imports.
2.  **Location**: 
    - Plugins: `domains/{domain}/plugins/`.
    - Models: `domains/{domain}/models/`. (Auto-discovered as metadata).
3.  **New Domain Hierarchy**: 
    - `domains/{name}/__init__.py`
    - `domains/{name}/models/`
    - `domains/{name}/plugins/`
4.  **Single-File Clean Architecture**: Each plugin MUST be contained within a single file. Logic, validation, and execution flow should NOT be split.
5.  **Sovereign Validation**: Use Pydantic models for API schemas and static methods in the `UserModel` for business validation.
6.  **Return Format**: Always return `{"success": bool, "data": ..., "error": ...}`.

### Communication & Events
**The Golden Guard**: Communication between domains is **STRICTLY EVENT-DRIVEN**.
- **No Cross-Imports**: Plugins MUST NOT import from other domains.
- **Event Standards**: 
    - Use `event_bus.publish(name, data)` for fire-and-forget side effects.
    - Use `event_bus.request(name, data)` ONLY for mandatory cross-domain queries.
    - Successful state changes (e.g., `user.created`) MUST be published.

### Tools & Discovery
1.  **Isolation**: Tools MUST NOT import or depend on other Tools.
2.  **Location**: `tools/{tool_name}/`.
3.  **Complex Tools**: Tools may include subfolders (e.g., `migrations/`, `configs/`). The entry point is ALWAYS `{tool_name}_tool.py` at the root of the tool folder.
4.  **Inheritance**: MUST inherit from `BaseTool`.
4.  **Documentation**: `get_interface_description()` MUST include exact method signatures for AI consumption.
5.  **Mandatory Context Check**: Read [AI_CONTEXT.md](../../../AI_CONTEXT.md). This file is the "Live Manual" generated by the Kernel.
6.  **Deep Inspection**: If a tool's behavior is unclear, view its implementation in `tools/{tool_name}/`.
7.  **Injection**: Request the tool by its name in your plugin's `__init__`.

### Advanced Standards
- **API Schema**: When using `http_server.add_endpoint`, always provide `request_model` and `response_model` using Pydantic classes from your domain models.
- **Side Effects**: Successful state changes MUST publish an event via `event_bus.publish("domain.event_name", data)`.
- **Tool Lifecycle**: 
    - `setup()`: Internal tool initialization.
    - `on_boot_complete(container)`: Actions that require the full system to be online.

### Lifecycle Protocol
1.  **Allocation Phase (`setup`)**: Tools MUST initialize resources here.
2.  **DI Phase (`__init__`)**: Plugins MUST only save tool references here. No logic.
3.  **Registration Phase (`on_boot`)**: Plugins MUST register routes and event listeners here.
4.  **Action Phase (`execute`)**: Standard entry point for business logic.
5.  **Orchestration Phase (`on_boot_complete`)**: Tools may interact with the full container here.

### Step-by-Step for Creating a Plugin
1.  **Identify Domain**: Determine the target domain.
2.  **Consult Context**: Read `AI_CONTEXT.md` for tool availability.
3.  **Create Model**: Define Pydantic schemas in `domains/{domain}/models/`.
4.  **Implement Plugin**: Create the class in a single file following the architecture rules.
5.  **Register & Execute**: In `on_boot`, register endpoints with schemas. In `execute`, follow Validate -> Process -> Act -> Respond.
